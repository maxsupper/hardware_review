> 适用场景：原理图/网表信号与芯片引脚功能的交叉验证

## 核心原则

**网名是标签，不是物理事实。先追踪，后对比。**

原理图中的网络名称是设计者手工输入的字符串，可能因版本迭代、复制粘贴、多人协作等原因与实际连接不一致。不能以网名作为判断依据。

## 正确流程

### 第一步：追踪物理连接

从接口侧出发，沿信号路径追踪到芯片引脚：

```
接口引脚 → 网表网络 → 中间器件（电阻/电平转换等）→ 芯片引脚
```

**工具**：使用 EDIF/网表文本解析，逐段追溯每个 `portRef`。

```
示例：
  J5.2 → net "UART3_TX" → U9.B1 → U9.A1 → net "UART3_TX_3V3" → U1000(GPIO2_A3_D_J)
```

**⛔ 跨无源器件追踪（2026-06-15 新增）**：

串联电阻/电感/磁珠会断开网名，必须跨器件继续追踪：

```
GMAC0_TXEN 信号：
  J1.56 → net "GMAC0_TXEN" → R11.&1
                               ↓ 跨电阻
  R11.&2 → net "GMAC0_TXEN_SOC" → U6.GPIO2_C0_d_H
```

**规则**：
1. 如果连接器引脚所在 net 未找到 U6 portRef，查该 net 上所有无源器件（RESISTOR/INDUCTOR/FERRITE）
2. 对每个无源器件，查其对端所在的 net → 递归搜索 U6 portRef
3. 最多递归 2 层（串联电阻→U6，串联电阻→电平转换器→U6）
4. 找到 U6 GPIO 后继续执行反向供电校验

**反例**：GMAC0_TXEN 仅搜了 "GMAC0_TXEN" net → 未找到 U6 → 标为 DEDICATED/3.3V ❌

### 连接器引脚输出格式

> ⚡ **强制检查（自学习 K-T005）**：连接器逐引脚追踪已在 3 个项目累计 571 引脚验证为最易遗漏的检查项。以下流程为强制检查，不可跳过。

每个连接器引脚必须包含以下 8 列，所有列必须有值：

| Pin | Net | Signal | Direction | Voltage | Domain | Source | Dest |
|-----|-----|--------|-----------|---------|--------|--------|------|
| 引脚编号 | 网络名 | 信号功能描述 | Output/Input/Bidirectional/PWR/GND | 实际电压值 | VCCIO域 | 源端IC+引脚 | 目的端IC+引脚 |

- Pin：连接器引脚号（如 1, A, B…）
- Net：EDIF 网表中的物理网络名
- Signal：该引脚承载的信号功能（如 UART3_TX, VCC5V, GND）
- Direction：信号方向（Output=SoC→外设, Input=外设→SoC, Bidirectional, PWR=电源输入, GND=地）
- Voltage：反向校验确认的 I/O 电压（如 3.3V, 1.8V），不得为 unknown
- Domain：该信号在 SoC 侧归属的 VCCIO 域（如 VCCIO3, EMMCIO_1V8）
- Source：驱动端 IC 位号 + 引脚（如 U6.GPIO2_C0_d_H）
- Dest：接收端 IC 位号或连接器标识（如 J1.Pin56）

### 第二步：确认芯片引脚功能

在芯片引脚复用表中查找该引脚的全部功能：

```
GPIO2_A3_D_J:
  Mux0: UART3_TX
  Mux1: FEPHY_LEDLINK
  Mux2: PWM0_CH2_M2
  ...
```

验证接口需要的功能是否存在于该引脚的复用选项中。

### 第三步：确认 VCCIO 域电压

从网表中追踪芯片供电引脚的实际电压：

```
VCCIO2_VCC_J → net "VCC_3V3" → 3.3V
VCCIO1_VCC_I → net "VCC_1V8" → 1.8V
```

根据芯片引脚的域后缀（_I/_J/_K 等）查表得电压。

### 第四步：反向验证标签

将前三步的物理事实与原理图标签对比，记录差异：

| 验证项 | 来源 | 方法 |
|--------|------|------|
| GPIO Bank 号 | 芯片引脚名 | 解析 portRef 中的 GPIOx_* 字段 |
| 功能正确性 | 芯片复用表 | 检查功能是否在 Mux 列表中 |
| 电压电平 | VCCIO 域供电 | 追溯供电网络 |
| 信号方向 | 中间器件方向 | 电平转换器方向/缓冲器方向 |

## 禁止事项

| ❌ 禁止 | ✅ 应做 |
|--------|--------|
| 以网名判断 GPIO Bank 号 | 追踪网表到芯片引脚，从引脚名提取 Bank |
| 以网名判断电压 | 追踪 VCCIO 域的实际供电网络 |
| 以网名判断功能 | 在芯片复用表中确认该功能存在 |
| 跳过中间器件直接推断 | 逐段追踪：接口→电阻→转换器→芯片 |
| 相信单一的 VCCIO 电压假设 | 逐个追踪每个 VCCIO 引脚的实际网络 |
| 只验证功能类别存在 | **验证编号/数值精度匹配**（如 CIF_D1 vs DATA0） |
| 有串联电阻就跳过 | 追踪电阻另一端连接的完整网络名 |
| 功率路径中途停止（见网名即结论） | 追到芯片电源引脚（VDD/VDDQ/VDD_CORE 等）才算终点（2026-06-01 新增） |
| 忽略磁珠/电感/分流器等功率无源器件 | FB/L/PR 也是中间器件，必须逐段追踪其两端（2026-06-01 新增） |
| 未穿透分流器（Kelvin Shunt） | 4-pin 分流器的电流引脚（&2/&4）和采样引脚（&1/&3）分别追踪（2026-06-01 新增） |

## 实战案例

> 以下案例来自某 RV1126B 项目的实际审查，用于演示方法论。具体芯片型号、引脚名、接口编号因项目而异，但追踪方法和判定逻辑是通用的。

### 案例：接口 "GPIO1-3V3" 的核实

**错误做法**：看到网名 `GPIO1-3V3` → 查 GPIO1 的 VCCIO1 = 1.8V → 结论 "3.3V 标错了"

**正确做法**：

```
1. 追踪物理连接：
   J3.5 → net "GPIO1-3V3" → U1000 portRef:
   "UART4_RX_M3  UART3_RTSN_M0  SDMMC0_CLK  GPIO2_A4_D_J"

2. 提取芯片引脚：GPIO2_A4_D_J
   GPIO Bank = GPIO2（不是 GPIO1！）
   域后缀 = _J

3. 查 VCCIO 域：
   VCCIO2_VCC_J → VCC_3V3 = 3.3V ✓

4. 对比标签：
   网名 GPIO"1"-3V3 → 实际 GPIO"2"_A4 → Bank 号错，电压对
```

### 案例：J3.9 "GPIO0-1V8" 的核实

```
1. 追踪：J3.9 → net "GPIO0-1V8" → U1000:
   "DSMC_INT2  UART3_CTSN_M1  PWM1_CH0_M1  SPI1_MISO_M2  VO_LCDC_CLK  GPIO5_D3_D_L"

2. 芯片引脚：GPIO5_D3_D_L
   GPIO Bank = GPIO5（不是 GPIO0！）
   域后缀 = _L → VCCIO5 = 1.8V

3. 电压 1.8V ✓，Bank 号错（GPIO0 → GPIO5）
```

### 案例：DVP VI0_DATA0/1 CIF 编号互换

```
J5.15 标号 "VI0_DATA0"
 → 追踪 EDN: J5.15 → net VI0_DATA0 → U1000 portRef:
   "SARADC1_IN0  ETH_TXD1_M0  SAI0_SDO2_M1  VI_CIF_D1_M0  ...  GPIO6_A1_D_M"
 → CIF 功能: VI_CIF_D1_M0（编号 D1 ≠ 标号 DATA0）
 → GPIO: GPIO6_A1

J5.20 标号 "VI0_DATA1"
 → 追踪 EDN: J5.20 → net VI0_DATA1 → U1000 portRef:
   "... VI_CIF_D0_M0 ... GPIO6_A0_D_M"
 → CIF 功能: VI_CIF_D0_M0（编号 D0 ≠ 标号 DATA1）
 → GPIO: GPIO6_A0
```

**发现**：D0 和 D1 的 CIF 功能编号互换了。仅检查"有 CIF 功能"不够，必须比对编号。

## 精确值校验规则（No-Shortcut Rule）

### 问题根源

只验证"功能类别存在"（有 CIF ✅）而未验证"编号精确匹配"（CIF_D1 ≠ DATA0）。

```python
# ❌ 浅层检查 — 仅验证类别存在
if 'CIF' in func:  →  "通过"
# 遗漏了: CIF_D1 vs 标号 DATA0 不一致

# ✅ 深层检查 — 验证编号精确匹配
cif_num = extract_number(cif_func)      # D1 → 1
label_num = extract_number(net_name)    # DATA0 → 0
if cif_num != label_num:  flag MISMATCH
```

### 适用场景

以下接口必须做**编号级别的精确校验**：

| 接口类型 | 需精确校验的编号 | 示例 |
|----------|------|------|
| DVP 并行视频 | DATA[0:15] ↔ CIF_D[0:15] | D0→D0, D1→D1, ... |
| SDMMC | DATA[0:3] ↔ SDMMC_D[0:3] | D0→D0 |
| SPI | CS[0:N] ↔ CSx | CS0→CS0 |
| I2C | SCL/SDA ↔ 同编号控制器 | I2C1_SCL + I2C1_SDA |
| UART | TX/RX ↔ 同编号控制器 | UART3_TX + UART3_RX |
| GPIO | Bank+Pin ↔ 一致 | GPIO2_A4, not GPIO1 |
| DDR | DQ[0:31] ↔ 顺序对应 | DQ0→DQL0 |

### 禁止事项（补充）

| ❌ 禁止 | ✅ 应做 |
|--------|--------|
| 仅验证功能类别（有 CIF） | **验证编号精确匹配**（CIF_D1 vs DATA0） |
| 仅验证"连着 SoC" | 验证连到了正确编号的 SoC 引脚 |
| 批量检查时跳过数值对比 | 对有编号的信号逐位编号对比 |

## 网表解析技巧

### EDIF 格式 portRef 提取

```python
# 提取网络中所有连接
portRef &5 (instanceRef J3)                     # 接口侧
portRef GPIO2_A4_D_J (instanceRef U1000)        # 芯片侧（多功能合并）
portRef UART4_RX_M3 UART3_RTSN_M0 ... U1000)    # 芯片侧（展开所有复用功能）
```

关键字段：
- 第一个标识符是**引脚编号**（如 `&5`, `A1`）
- 多功能引脚包含管道符分隔的功能列表
- GPIO 名格式：`GPIO{Bank}_{Pin}_{Dir}_{Domain}`

### GPIO 域后缀与 VCCIO 映射

| 后缀 | VCCIO 域 | 需从网表确认实际电压 |
|:---:|:--------:|:---:|
| `_H` | PMUIO | 追踪 PMUIO_VCC 网络 |
| `_I` | VCCIO1 | 追踪 VCCIO1_VCC 网络 |
| `_J` | VCCIO2/3 | **注意：同后缀对应两个不同域** |
| `_K` | VCCIO4 | 追踪 VCCIO4_VCC 网络 |
| `_L` | VCCIO5 | 追踪 VCCIO5_VCC 网络 |
| `_M` | VCCIO6 | 追踪 VCCIO6_VCC 网络 |
| `_F` | MIPI PHY | 追踪 MIPI_AVDD 网络 |

**特别注意**：`_J` 后缀同时对应 VCCIO2 和 VCCIO3，必须从网表中追踪具体引脚所在 BANK 的实际供电网络，不能直接假设电压。

## 自动化脚本自我审查规则（Script Self-Audit）

### 问题根源

自动化检查脚本自身的**截断、简化、假设**是漏检的首要原因。

### 案例 1：`net_body` 行数截断 → I2C2 上拉漏检

```python
# ❌ 有截断风险的写法
for j in range(i, min(len(lines), i+50)):  # 固定 50 行
    body.append(lines[j])
    ...
# 网络较长 → U1000 portRef 在第 55 行 → 被截断 → has_u1000=False
# → 误判为 "no U1000, assumed external" → 跳过上拉检查 !!
```

```python
# ✅ 正确的写法 — 扫描到网络结束符
depth = 0
for j in range(i, len(lines)):
    body.append(lines[j])
    if lines[j].strip().startswith('(net ') and j > i:  # 遇到下一个 net
        break
    if lines[j].strip() == ')':
        depth -= 1
        if depth <= 0:
            break
```

### 案例 2：仅验证功能类别 → DVP D0/D1 编号漏检

```python
# ❌ 浅层检查
if 'CIF' in func:  # 有 CIF 功能 → 通过
    pass          # 未比较 CIF_D1 vs DATA0 !!
```

```python
# ✅ 深层检查
cif_num = extract_number(cif_func)    # D1 → 1
label_num = extract_number(sig_name)  # DATA0 → 0
assert cif_num == label_num
```

### 案例 3：只搜 portRef 已连接引脚 → U3200 ZQ 悬空漏检

```python
# ❌ 只搜 portRef（网络连接的引脚）
ports = re.findall(r'portRef\s+(\S+)\s+\(instanceRef U3200\)', content)
# → 结果: VSS/ZQ_U(DDP) 等已连接引脚
# → ZQ 引脚悬空 → 没有 portRef → 永远搜不到！
# → 搜到 VSS/ZQ_U 含 "ZQ" 子串 → 误判为 ZQ → 停止
```

```python
# ✅ 对比 portInstance（符号定义） vs portRef（网络连接）
# 1. 从 instance 定义提取全部引脚
inst_pins = re.findall(r'portInstance\s+(\S+)', instance_body)
# → ['DML_n', 'DQSL_P', ..., 'ZQ', ..., 'VSS/ZQ_U(DDP)']
# 2. 从网络提取已连接的引脚
wired_pins = re.findall(r'portRef\s+(\S+)\s+\(instanceRef U3200\)', content)
# → ['DML_n/DBIL_n', ..., 'VSS/ZQ_U(DDP)']  # 没有 ZQ！
# 3. 对比 → ZQ 在 inst_pins 但不在 wired_pins → 悬空！
floating = inst_pins - wired_pins
```

### 悬空引脚检测规则

> ⚠ **2026-05-27 修订**：增加"按芯片，不按引脚类型"原则。原规则以引脚功能为枚举维度，但正确的枚举维度是**芯片实例 × 引脚功能**。仅按功能枚举会导致对非主芯片（DRAM、PMIC、eMMC 等）漏检。

#### 芯片范围规则（新增，不可跳过）

以下芯片必须执行全量 portInstance-vs-portRef 对比：

| 芯片类型 | 判定门槛 | 未 diff 则 |
|----------|:---:|------|
| **主 SoC** | 必须 | 漏检 U1000 功能引脚 |
| **DDR 颗粒**（每颗独立） | 必须 | 漏检 ZQ/VREF/RESET 悬空 |
| **PMIC** | 必须 | 漏检电源输出/使能悬空 |
| **eMMC / Flash** | 必须 | 漏检 CMD/CLK/RST 悬空 |
| **PHY 芯片**（ETH/MIPI） | 建议 | 漏检关键信号 |
| **电平转换器** | 建议 | 漏检 VCCA/VCCB/OE |

**执行流程**（对每个芯片独立执行）：

```
for each 关键芯片:
    1. 从 instance 定义提取 portInstance 全量列表 → ALL_PINS
    2. 从网表提取 portRef 全量列表（已连接引脚） → WIRED_PINS
    3. FLOATING = ALL_PINS − WIRED_PINS
    4. 对 FLOATING 中的每个引脚，按功能类型判级：
       功能引脚(ZQ/VREF/RESET/CKE…) → 🔴 CRITICAL
       电源引脚(VDD/VDDQ)            → 🔴 CRITICAL
       GND 引脚(VSS/VSSQ)            → 🟡 WARNING（可能通过基板连接）
       NC 引脚                       → 🟢 OK
       专用功能引脚(CLK/CMD/DQ…)     → 🔴 CRITICAL
```

#### 实战教训

U3200 (DDR4) 的 `ZQ` 引脚（pin 55）在 portInstance 中定义，但不在任何 net 的 portRef 中，导致悬空。初检仅验证了 SoC 端的 DDR_RZQ 和已连接的 R3207(VSS/ZQ)，未对 U3200 执行全量 diff，故漏检。

**漏检根因**：不是不知道要查 ZQ——知道、也查了 R3210 和 R3207。而是查了"已连接的那几个 ZQ 信号"就停了，没有先做全量 diff 再分类。

#### 原规则（保留，按引脚功能分类）

对于以下关键功能引脚，必须检查符号定义与网络连接的一致性：

| 引脚类型 | 必须连接 | 检测方法 |
|----------|:---:|------|
| DDR ZQ | 240Ω→GND | portInstance 有 ZQ 且 portRef 存在且接电阻 |
| DDR VREF | 接 VREF 电路或 GND | portInstance 有 VREF 且 portRef 存在 |
| DDR RESET | 接 SoC + 上拉 | portRef 存在 |
| 电源引脚(VDD/VDDQ) | 接电源网络 | portRef 存在 |
| GND 引脚(VSS/VSSQ) | 接 GND | portRef 存在 |
| NC 引脚 | 可悬空 | 无 portRef 属正常 |

### 脚本自审清单

写脚本时，对每个自动化检查函数执行以下自审：

| 审查项 | 问题 | 修复 |
|--------|------|------|
| **边界条件** | 固定行数截断是否遗漏数据？ | 改为内容感知的边界检测 |
| **正则跨网** | `re.DOTALL` 是否跨多个 net？ | 先提取 net body，再在 body 内匹配 |
| **分支路径** | 信号经串联电阻是否追踪到另一端？ | 追踪电阻另一端的网络 |
| **比较精度** | 只验证类别还是验证编号？ | 逐位比较编号 |
| **"assumed OK"** | 未找到数据时默认通过还是上报？ | 默认上报异常，不假设正常 |
| **portRef vs portInstance** | 只搜已连接引脚 → 悬空脚漏检 | 提取 portInstance 全量列表与 portRef 对比 |
| **子串匹配** | "ZQ" 匹配到 "VSS/ZQ_U" → 假阳性 | 精确匹配端口全名 |

### 错误处理规则

```python
# ❌ 沉默跳过
if not found:
    return  # 默认正常 ← 错误！

# ✅ 显式上报
if not found:
    add_issue('WARNING', f'{name}: trace incomplete - manual check required')
    return
```

### 验证策略层级

```
Level 1: 连接存在性 (has portRef?)      ← 基础
Level 2: 功能类别匹配 (has CIF/UART?)   ← 中层
Level 3: 编号精确匹配 (CIF_D1 == DATA1?) ← 深层 ← 不可跳过！
Level 4: 电气匹配 (VCCIO电压? 上拉?)     ← 完整
```

**Level 1-2 通过 ≠ 实现正确。Level 3-4 是必检项。**

## 检查员自审清单（Inspector Self-Audit）

> 适用场景：完成任何硬件检查后，交付报告前，逐项自审。不依赖自动化脚本，人工检查也必须执行。

### 一、枚举先行原则（Enumerate Before Evaluate）

**禁止**边发现边判断。必须先拉清单，后逐项核验。

```
❌ 错误做法：发现 I2C0 有上拉 → 产生 "I2C 应该都有" 的印象 → 跳过我 I2C1/I2C2

✅ 正确做法：
  第一步：拉清单（只发现，不判断）
    本设计共有 3 组 I2C：I2C0 / I2C1 / I2C2
    每组 2 个信号 → 共 6 个信号需要检查上拉

  第二步：逐项核验（按清单依次，不可跳跃）
    I2C0_SCL → 读 net body → 找到 R2308 → ✅
    I2C0_SDA → 读 net body → 找到 R2307 → ✅
    I2C1_SCL → 读 net body → 无 R → ❌ CRITICAL
    I2C1_SDA → 读 net body → 无 R → ❌ CRITICAL
    I2C2_SCL → 读 net body → 无 R → ❌ CRITICAL
    I2C2_SDA → 读 net body → 无 R → ❌ CRITICAL
```

**适用接口类型**：I2C、UART、SPI、SDMMC、DVP、MIPI 等一切多实例总线。

### 二、逐 net body 硬读（Hard-Read Net Body）

每个待检信号必须实际打开 net body，不可凭经验推断。

| ❌ 凭经验跳过 | ✅ 硬读确认 |
|---|---|
| "连着 SoC，连着 J3，应该没问题" | 打开 net body，确认 portRef 列表中有 `Rxxxx` |
| "I2C 一般都有上拉" | 逐总线、逐信号确认 |
| net body 只有两行就不读了 | 两行也必须打开读 |

### 三、自审门禁（Go/No-Go Gates）

> **完整的 G0-G5 门禁定义见 [hardware-reviewer.md](./hardware-reviewer.md) §⛔ 启动门禁：Step 0 及 §Self-Audit Gates**
>
> 本章保留的是方法论层面的检查清单（脚本自审清单、枚举先行、硬读 net body），与门禁系统互补但不重复。所有门禁检查项的唯一定义在 hardware-reviewer.md，如有不一致以 hardware-reviewer.md 为准。

### 四、经典漏检模式速查

### 四、经典漏检模式速查

以下组合极易被跳过，检查时必须额外警戒：

| 漏检模式 | 典型场景 | 为什么容易被跳 |
|----------|----------|---------------|
| **首项正确 → 尾项跳过** | I2C0 有上拉 → 判定 I2C1/I2C2 也有 | 正例产生认知惯性 |
| **低优先级标签 → 降低检查深度** | "只是 debug 口" "预留接口" | 人为降级 |
| **简单总线 → 轻视** | I2C 只有两根线 → "不可能出错" | 低估风险 |
| **大量重复 → 检查疲劳** | DVP 16 条 DATA 线 | 第 5 条后开始走捷径 |
| **net body 短 → 不打开** | 只有 J3→U1000 直连 | 臆断"没有中间器件" |
| **🔴 规则文档替代手册原文** | 规则文档写了"MS4553S 自动方向检测"→ 不再查阅 MS4553S.pdf | 二手信息不会被质疑，且遗漏封装不匹配(§4.11) |
| **🔴 手册可得但未读取** | PDF 在本地 D:\Refbook，但检查时未打开 | "已有描述"产生"已确认"的错觉(违反 G5-2) |
| **🔴 推论链当事实写入** | 看到 `_J` 后缀→知道 VCCIO2/3 分叉→联想规则文档中的标注错误案例→推断"此处标注可能也不对"→在未核实 EDN 的情况下写入"标注含3.3V" | 规则文档的案例产生锚定效应，推论未经 EDN 验证直接成为"结论" |

### 五、纠正措施

当自审发现漏检时：

1. **立即停止当前判定**，不要基于已有印象补充结论
2. **回到枚举清单**，对该类接口从头逐项重新检查
3. **更新检查结果**，明确标注"补充检查"及遗漏原因
4. **记录漏检模式**，下次同类接口检查前预先警戒

## 输出要求：VCCIO 域→电压映射表（数据交接）

> ⛔ **强制执行**：完成本文件的接口→Bank→VCCIO→电压追踪后，必须产出两份文件：
> 1. **临时文件** `.sisyphus/temp/vccio.msg` — 紧凑 minified JSON，供后续步骤机器读取（格式见 `hardware-reviewer.md` §前后级数据交接）
> 2. **证据文件** `.sisyphus/evidence/vccio_map.json` — 人类可读 JSON，供报告引用
>
> 后续步骤**禁止**重新到 EDN 追踪电压，一律读取 `.sisyphus/temp/vccio.msg`。

### JSON Schema

```json
{
  "source": "引脚核对规则.md Step 1-3 追踪输出",
  "generated_at": "ISO8601",
  "edn_file": "原始 EDN 文件名",
  "vccio_domains": {
    "PMUIO": {
      "suffix": "_H",
      "power_pins": ["PMUIO0_VCC3V3", "PMUIO1_VCC"],
      "power_net": "VCC3V3_SYS",
      "voltage": 3.3,
      "voltage_source": "PMIC_3V3.Buck1",
      "evidence_edn_line": 1234
    },
    "VCCIO1": {
      "suffix": "_I",
      "power_pins": ["VCCIO1_VCC_I"],
      "power_net": "VCC_1V8",
      "voltage": 1.8,
      "voltage_source": "PMIC_1V8.LDO",
      "evidence_edn_line": 567
    }
  },
  "pins": [
    {
      "interface": "Jx.y",
      "net_name": "UART3_TX",
      "chip_pin": "GPIO2_A3_D_J",
      "gpio_bank": "GPIO2",
      "vccio_domain": "VCCIO2",
      "vccio_suffix": "_J",
      "voltage": 3.3,
      "evidence_edn_line": 890
    }
  ]
}
```

### 输出步骤

1. 按 `§GPIO 域后缀与 VCCIO 映射` 表建立 `{后缀 → VCCIO 域名}` 映射
2. 对每个 VCCIO 域，在 EDN 中追踪其电源引脚（如 `VCCIO2_VCC_J`）所在的 net
3. 沿 net 追踪到供电来源（PMIC/LDO/Buck 的输出），提取实际电压值
4. 对每个已追踪的接口信号，记录其所属芯片引脚、VCCIO 域后缀、最终电压
5. 写入 `.sisyphus/evidence/vccio_map.json`

### 验证

- [ ] `vccio_domains` 表覆盖 EDN 中存在的所有 VCCIO 域后缀
- [ ] 每个域的 `voltage` 值为实际追踪结果，非假设值
- [ ] 每条关键结论的 `evidence_edn_line` 可回溯到 EDN 原文
